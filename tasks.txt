[DONE] 2. Swiss algorithm — properly. The previous "done" claim was wrong:
   stress-testing the n=16, 12-round white-wins config produced 21 emergency
   byes across 8 rounds where players were dropped from pairings entirely.
   Four bugs found and fixed in pairing.py:
     a. _backtrack's worst-case baseline was list(top) instead of
        list(group), so when no permutation beat the baseline, bottom-half
        players silently vanished from the output.
     b. The floater cascade only carried ONE player between score groups —
        when a whole group's members had played each other, the rest got
        emergency byes instead of floating down together.
     c. _backtrack only permuted the bottom half, so within a group it
        could never pair top[0] with top[1] even when that was the only
        legal pairing. Replaced with _max_matching: a real maximum-matching
        search over the full group, with _greedy_matching as fallback for
        groups >14 players.
     d. Per-group commitment could leave emergency byes when a tournament-
        wide pairing existed. Added step 6: if the score-group cascade
        leaves anyone unmatched, retry _pair_group on the whole pool. Only
        fires when the cascade fails, so normal Swiss quality is preserved.

   Verified: n=3 through n=32, up to 12 rounds, white-wins and all-draws.
   Zero rematches, zero missing players, zero emergency byes except when
   logically impossible (n=3 with 5 rounds — only 3 unique pairs exist).
   Brute-force optimality check (check_optimality.py) confirms no emergency
   bye is emitted when a perfect matching exists. ~14ms total for n=32, 12
   rounds. Test files: test_swiss.py, check_optimality.py, test_quality.py.

[DONE] 4. Buchholz + Sonneborn-Berger tiebreaks.

   Schema: added `sonneborn_berger REAL NOT NULL DEFAULT 0` to the players
   table. Added a `_migrate` helper in database.py that does an idempotent
   ADD COLUMN for existing dev databases (guarded by PRAGMA table_info, so
   re-running is safe). Verified the migration preserves existing player
   rows (score, buchholz untouched; new column defaults to 0).

   Computation: renamed _recompute_buchholz → _recompute_tiebreaks in
   tournament.py. Now computes both tiebreaks in one pass over confirmed
   matches per player. Same trigger as before (every _finalize_match call),
   so the recompute cadence is unchanged.

   Definitions used:
     - Buchholz: sum of distinct opponents' final scores. Byes excluded.
     - Sonneborn-Berger: sum over each game of (opponent_score *
       game_weight), where weight = 1.0 for a win, 0.5 for a draw, 0.0
       for a loss. Byes excluded (no opponent score to weight).
   This is the FIDE-standard SB. If the club prefers the "Median Buchholz"
   or "Cumulative" variants, they're additional columns added later.

   Standings order: list_players now sorts by
     score DESC, buchholz DESC, sonneborn_berger DESC, elo DESC, name ASC
   So Buchholz is the first tiebreak, SB is the second. This matches FIDE's
   most common Swiss order.

   UI:
     - host.html standings table: added an SB column next to BH. Both have
       title= tooltips explaining what they are.
     - projector.html leaderboard: the existing BH chip became
       "BH X · SB Y" on the same row.

   Tests: test_tiebreaks.py verifies four cases —
     a. Classic 4-player round-robin where SB matches textbook values
        (Alice 3.0 / Bob 1.0 / Carol 0.0 / Dave 0.0).
     b. Two players tied on score+BH; SB equal too because their
        opponents ended on equal scores. Documented in the test.
     c. Two players on equal score but Alice beat stronger opponents;
        SB correctly orders Alice first.
     d. A bye contributes 0 to SB and 0 to BH (FIDE standard).

   Verified end-to-end: full report → confirm → snapshot pipeline returns
   sonneborn_berger in WS broadcasts. The original task #3 e2e test still
   passes against the task #4 codebase, no regressions.

[DONE] 6. Bye handling polish. The old rule in pair_swiss step 1 (and the
   identical block in pair_random) was: pick the lowest-scored player who
   hasn't had a bye; if everyone has had one, pick candidates[0]. The
   fallback ignored bye counts entirely, so in small fields over many
   rounds the same player could collect a second bye before others got
   their first second bye — bye counts could drift by 2+.

   Fix: new helpers _bye_count and _select_bye_recipient in pairing.py.
   Recipient is now chosen by (bye_count ASC, score ASC, elo ASC) — single
   ordering that subsumes both the "anyone without a bye yet" pass and the
   "lowest score otherwise" fallback. Guarantees the field's bye counts
   stay within ±1 of each other for as long as legal pairings exist.

   Round 1 behavior is unchanged (everyone is at bye_count=0, so it falls
   through to lowest-score-then-elo, same as before). Same one-line change
   applied to pair_random for consistency — same bug lived there.

   Verified:
     - Unit checks on _select_bye_recipient: skips bye'd players when
       others haven't had one; picks lowest score among tied bye counts;
       won't hand out bye k+1 to a player while anyone in the field still
       has bye count < k.
     - n=5 across 5 rounds: each player gets exactly one bye, in turn.
     - Regression: n=4 through n=16 over min(5, n-1) rounds (within the
       unique-pairing budget) — zero rematches, zero missing players,
       expected bye count per round. Task #2's invariants are intact.

   Out of scope: when the field is logically exhausted (n=3 after 3
   rounds, n=4 after 3 rounds, n=5 after 5 rounds), the algorithm correctly
   emits emergency byes for everyone — same behavior as before #6. If the
   host should be told "tournament cannot continue" at that point instead
   of producing all-bye rounds, that's a tournament.py UX change, not a
   pairing.py change. Logged for later.

[DONE] 8. Round timer on projector.

   The brief called this "pure frontend" but it isn't — the state snapshot
   from get_state_snapshot didn't expose any per-round timestamp, so a
   client-only timer would either reset on every reconnect or count from
   page load (meaning a viewer who opens the projector mid-round would see
   the wrong number). Surfaced the existing rounds.created_at instead, no
   schema migration needed.

   Backend (tournament.py): get_state_snapshot now reads the current
   round's created_at and returns it as `round_started_at`. SQLite's
   CURRENT_TIMESTAMP yields "YYYY-MM-DD HH:MM:SS" in UTC with a space
   separator; converted to "YYYY-MM-DDTHH:MM:SSZ" before sending. V8
   tolerates the space form but the ECMAScript spec only guarantees
   Date.parse on the T-separated ISO 8601 variant — Safari historically
   returned NaN on the space form. The replace + Z append is the cheapest
   way to be cross-browser safe. None when no round has started yet.
   Existing snapshot consumers (host.html, player.html) ignore unknown
   keys — no regressions.

   Frontend (projector.html):
     - New "Elapsed" stat block in the header, hidden when current_round
       is 0 or the timestamp is null/unparseable.
     - updateRoundStart() called from render() reseats the anchor from
       every state snapshot, so a new round resets the timer to 0:00 the
       instant the new round_started_at arrives via WS.
     - 1Hz setInterval calls tickElapsed() which reads Date.now() minus
       the anchor — the formatter is integer ops plus one textContent
       write, cheap enough that there's no point throttling further.
     - formatElapsed: m:ss under an hour, h:mm:ss past one. Matches a
       sports clock — projector view, drama, etc.
     - Defensive: isNaN(Date.parse(...)) check hides the stat instead of
       crashing if the server ever sends garbage. Won't happen with the
       current backend, but the projector is the screen people SEE.

   CSS (projector.css): #proj-elapsed gets font-variant-numeric:
   tabular-nums so the seconds digits don't jiggle as glyph widths change
   each tick. Same trick already used on .rank-num and .rank-score.

   Verified:
     a. End-to-end test: created a 3-player onsite tournament, snapshot
        before round 1 had round_started_at = None (timer hidden); after
        start_next_round the snapshot returned an ISO-T string within
        1s of wall clock.
     b. Round transition: completed round 1 (report + confirm), started
        round 2 after a 2-second sleep; the new round_started_at was
        exactly 2s ahead of round 1's. The projector therefore resets to
        0:00 on round change, which is the right behavior.
     c. JS Date.parse round-trip: 2026-05-18T05:48:17Z parses correctly
        in V8 (test via `node -e`).
     d. node --check on the extracted projector.html script block passes.
     e. Python compile-check on tournament.py passes.
     f. State snapshot still returns tournament, players, current_matches,
        events as before — round_started_at is purely additive.

   Out of scope and logged for later: a *countdown* timer (round time
   limit, with an alarm when it hits zero) would need a per-tournament
   round_length_seconds setting, host UI to configure it, and a
   completion event on the WS broadcast. The brief just said "round
   timer", and an elapsed-time counter is what's honest about what we
   know without inventing config. If the club wants countdown, that's a
   follow-up task.

Verified vs. the original pair_swiss (A/B comparison, "white always wins"
stress test):
   - Old: produced duplicate pairings (rematches) under stress — 39 dupes
     in a 12-round 16-player simulation.
   - Intermediate (the version that was claimed "done"): zero rematches
     but achieved this by silently dropping players from rounds and
     emitting emergency byes — 21 emergency byes plus 8 rounds with
     missing players in the same 12-round 16-player simulation. The
     "zero duplicate pairings" claim was technically true but misleading;
     the algorithm was just refusing to pair instead.
   - Current (after #2 rewrite): zero duplicates AND zero emergency byes
     AND zero missing players across all tested configs (n=3 through
     n=32, up to 12 rounds, white-wins and all-draws scenarios). When an
     emergency bye is genuinely unavoidable (n=3 over many rounds), the
     algorithm distributes byes fairly.
