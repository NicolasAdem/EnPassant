[BUG] A1. host_token leak via WebSocket broadcasts.
  File: app/routers/api.py:62
  _broadcast_state() sends svc.get_state_snapshot() without scrubbing
  tournament.host_token, while HTTP state and the initial WS state
  explicitly scrub it. Any projector/player WS subscriber receives the
  host token after the next broadcast and can then call host-only
  endpoints.
  Fix: scrub host_token in _broadcast_state() the same way the initial
  WS state and HTTP state already do. Centralize the scrub so the three
  paths can't diverge again.
  Test to add: a WS subscriber on a non-host channel never sees
  host_token in any frame, including post-broadcast.

[BUG] A2. XSS via player name in host.html inline onclick.
  File: templates/host.html:245
  Player names are interpolated into an inline onclick after EP.escape().
  A name like  );alert(1);//  becomes &#39;);alert(1);// in HTML, which
  the browser decodes back to ' inside the event handler, breaking out
  of the string and executing script on the host dashboard. The attack
  vector is real: any joining player picks their own name.
  Fix: remove the inline onclick. Bind the handler in JS with
  addEventListener, keep the name in a data-* attribute or look it up
  by player id from state. The HTML-escape-then-decode trap goes away
  once the value never appears inside an attribute literal.
  Test to add: render host.html with a player named  );alert(1);//
  and confirm no script executes (smoke test or a regex check that
  no player-name interpolation appears inside an attribute value).

[BUG] A3. Disputed matches can bypass host resolution.
  File: app/services/tournament.py:304
  report_result() rejects only confirmed and bye, so a disputed match
  can be overwritten back to 'reported'. confirm_result() can then
  finalize it, bypassing the /matches/{mid}/resolve flow that Task 12
  claimed was the only path out of dispute.
  Fix: report_result() must also reject status='disputed'. Once a match
  is disputed, only the host's resolve endpoint can move it.
  Test to add: report -> dispute -> re-report attempt must 4xx (or
  whatever the existing rejection shape is). Confirm flow is closed
  until resolve runs.

[BUG] B1. set_match_table accepts byes.
  File: app/services/tournament.py:280
  Task 7 invariant: "Byes never get a table_number." set_match_table()
  only checks match id / tournament id, then updates ANY match including
  status='bye'. A host POST to /matches/{bye_mid}/table with
  {"table_number": 7} succeeds.
  Fix: reject status='bye' in set_match_table() with a clear error
  (4xx at the API layer). The api.py wrapper at
  POST /api/tournaments/{tid}/matches/{mid}/table should surface the
  rejection.
  Test to add: end-to-end — onsite tournament, start round, attempt to
  set a table on the bye match, expect 4xx and DB unchanged.

[BUG] B2. Swiss bye-recipient chosen before pool pairability check.
  File: app/services/pairing.py:238
  Task 2 invariant: "zero emergency byes when a perfect matching exists."
  Codex repro: with 5 players, past pairs
    {p00-p02, p01-p03, p03-p00, p01-p04, p00-p04, p02-p01}
  and scores p00=3, p01=1.5, p02=1.5, p03=2, p04=1, pair_swiss returns
  p04-p03 plus EMERGENCY BYES for p00, p02, p01. A valid one-bye round
  exists: bye p02, pair p00-p01 and p03-p04.
  Root cause: bye recipient is chosen by the Task 6 selector
  (bye_count, score, elo) without considering whether the remaining
  n-1 pool admits a perfect matching against the rematch graph.
  Fix sketch: when n is odd, try each candidate bye recipient in the
  Task 6 priority order, and accept the first whose REMOVAL leaves the
  remaining pool with a perfect matching (via _max_matching). Fall
  through to the current behavior only if no candidate works.
  Test to add: the Codex repro case in test_swiss.py, plus a
  regression that the Task 6 bye-fairness ordering is preserved
  whenever the priority-1 candidate is feasible.

[BUG] B3. Bye score updates skip _recompute_tiebreaks.
  Files: app/services/tournament.py:197 and app/services/tournament.py:260
  Task 4 invariant: tiebreaks are recomputed on every finalize. But
  the bye-score paths bump the bye'd player's score without calling
  _recompute_tiebreaks. Any previous opponent of the bye'd player has
  stale Buchholz/SB until some later non-bye match finalizes for that
  opponent.
  Fix: call _recompute_tiebreaks at both bye-update sites. Same trigger
  as _finalize_match.
  Test to add: 4-player Swiss, round 1 normal, round 2 awards a bye
  to a player whose round-1 opponent has confirmed games; verify the
  opponent's Buchholz/SB reflect the bye'd player's new score
  immediately after the bye finalizes, not only after a later match.

[BUG] B4. report/confirm ignore {tid} in path.
  Files: app/routers/api.py:155 and app/routers/api.py:167
  The report and confirm routes look up the match by match_id only,
  ignoring the {tid} segment. Posting a valid match/player pair under
  the WRONG tournament id mutates the real match but then broadcasts
  the wrong tournament's state — connected clients see desynced data,
  and the audit log shows the action attributed to the wrong
  tournament.
  Fix: verify the match's tournament_id matches the path {tid} before
  mutating. Reject with 404 (or 400 — match your existing convention)
  on mismatch.
  Test to add: report endpoint called with a real match_id under a
  different tournament's tid returns 4xx and does not mutate.
