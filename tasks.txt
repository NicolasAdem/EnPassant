[DONE] 2. Swiss algorithm — properly. The previous "done" claim was wrong:
   stress-testing the n=16, 12-round white-wins config produced 21 emergency
   byes across 8 rounds where players were dropped from pairings entirely.
   Four bugs found and fixed in pairing.py:
     a. _backtrack's worst-case baseline was list(top) instead of
        list(group), so when no permutation beat the baseline, bottom-half
        players silently vanished from the output.
     b. The floater cascade only carried ONE player between score groups —
        when a whole group's members had played each other, the rest got
        emergency byes instead of floating down together.
     c. _backtrack only permuted the bottom half, so within a group it
        could never pair top[0] with top[1] even when that was the only
        legal pairing. Replaced with _max_matching: a real maximum-matching
        search over the full group, with _greedy_matching as fallback for
        groups >14 players.
     d. Per-group commitment could leave emergency byes when a tournament-
        wide pairing existed. Added step 6: if the score-group cascade
        leaves anyone unmatched, retry _pair_group on the whole pool. Only
        fires when the cascade fails, so normal Swiss quality is preserved.

   Verified: n=3 through n=32, up to 12 rounds, white-wins and all-draws.
   Zero rematches, zero missing players, zero emergency byes except when
   logically impossible (n=3 with 5 rounds — only 3 unique pairs exist).
   Brute-force optimality check (check_optimality.py) confirms no emergency
   bye is emitted when a perfect matching exists. ~14ms total for n=32, 12
   rounds. Test files: test_swiss.py, check_optimality.py, test_quality.py.

[DONE] 4. Buchholz + Sonneborn-Berger tiebreaks.

   Schema: added `sonneborn_berger REAL NOT NULL DEFAULT 0` to the players
   table. Added a `_migrate` helper in database.py that does an idempotent
   ADD COLUMN for existing dev databases (guarded by PRAGMA table_info, so
   re-running is safe). Verified the migration preserves existing player
   rows (score, buchholz untouched; new column defaults to 0).

   Computation: renamed _recompute_buchholz → _recompute_tiebreaks in
   tournament.py. Now computes both tiebreaks in one pass over confirmed
   matches per player. Same trigger as before (every _finalize_match call),
   so the recompute cadence is unchanged.

   Definitions used:
     - Buchholz: sum of distinct opponents' final scores. Byes excluded.
     - Sonneborn-Berger: sum over each game of (opponent_score *
       game_weight), where weight = 1.0 for a win, 0.5 for a draw, 0.0
       for a loss. Byes excluded (no opponent score to weight).
   This is the FIDE-standard SB. If the club prefers the "Median Buchholz"
   or "Cumulative" variants, they're additional columns added later.

   Standings order: list_players now sorts by
     score DESC, buchholz DESC, sonneborn_berger DESC, elo DESC, name ASC
   So Buchholz is the first tiebreak, SB is the second. This matches FIDE's
   most common Swiss order.

   UI:
     - host.html standings table: added an SB column next to BH. Both have
       title= tooltips explaining what they are.
     - projector.html leaderboard: the existing BH chip became
       "BH X · SB Y" on the same row.

   Tests: test_tiebreaks.py verifies four cases —
     a. Classic 4-player round-robin where SB matches textbook values
        (Alice 3.0 / Bob 1.0 / Carol 0.0 / Dave 0.0).
     b. Two players tied on score+BH; SB equal too because their
        opponents ended on equal scores. Documented in the test.
     c. Two players on equal score but Alice beat stronger opponents;
        SB correctly orders Alice first.
     d. A bye contributes 0 to SB and 0 to BH (FIDE standard).

   Verified end-to-end: full report → confirm → snapshot pipeline returns
   sonneborn_berger in WS broadcasts. The original task #3 e2e test still
   passes against the task #4 codebase, no regressions.

[DONE] 6. Bye handling polish. The old rule in pair_swiss step 1 (and the
   identical block in pair_random) was: pick the lowest-scored player who
   hasn't had a bye; if everyone has had one, pick candidates[0]. The
   fallback ignored bye counts entirely, so in small fields over many
   rounds the same player could collect a second bye before others got
   their first second bye — bye counts could drift by 2+.

   Fix: new helpers _bye_count and _select_bye_recipient in pairing.py.
   Recipient is now chosen by (bye_count ASC, score ASC, elo ASC) — single
   ordering that subsumes both the "anyone without a bye yet" pass and the
   "lowest score otherwise" fallback. Guarantees the field's bye counts
   stay within ±1 of each other for as long as legal pairings exist.

   Round 1 behavior is unchanged (everyone is at bye_count=0, so it falls
   through to lowest-score-then-elo, same as before). Same one-line change
   applied to pair_random for consistency — same bug lived there.

   Verified:
     - Unit checks on _select_bye_recipient: skips bye'd players when
       others haven't had one; picks lowest score among tied bye counts;
       won't hand out bye k+1 to a player while anyone in the field still
       has bye count < k.
     - n=5 across 5 rounds: each player gets exactly one bye, in turn.
     - Regression: n=4 through n=16 over min(5, n-1) rounds (within the
       unique-pairing budget) — zero rematches, zero missing players,
       expected bye count per round. Task #2's invariants are intact.

   Out of scope: when the field is logically exhausted (n=3 after 3
   rounds, n=4 after 3 rounds, n=5 after 5 rounds), the algorithm correctly
   emits emergency byes for everyone — same behavior as before #6. If the
   host should be told "tournament cannot continue" at that point instead
   of producing all-bye rounds, that's a tournament.py UX change, not a
   pairing.py change. Logged for later.

[DONE] 8. Round timer on projector.

   The brief called this "pure frontend" but it isn't — the state snapshot
   from get_state_snapshot didn't expose any per-round timestamp, so a
   client-only timer would either reset on every reconnect or count from
   page load (meaning a viewer who opens the projector mid-round would see
   the wrong number). Surfaced the existing rounds.created_at instead, no
   schema migration needed.

   Backend (tournament.py): get_state_snapshot now reads the current
   round's created_at and returns it as `round_started_at`. SQLite's
   CURRENT_TIMESTAMP yields "YYYY-MM-DD HH:MM:SS" in UTC with a space
   separator; converted to "YYYY-MM-DDTHH:MM:SSZ" before sending. V8
   tolerates the space form but the ECMAScript spec only guarantees
   Date.parse on the T-separated ISO 8601 variant — Safari historically
   returned NaN on the space form. The replace + Z append is the cheapest
   way to be cross-browser safe. None when no round has started yet.
   Existing snapshot consumers (host.html, player.html) ignore unknown
   keys — no regressions.

   Frontend (projector.html):
     - New "Elapsed" stat block in the header, hidden when current_round
       is 0 or the timestamp is null/unparseable.
     - updateRoundStart() called from render() reseats the anchor from
       every state snapshot, so a new round resets the timer to 0:00 the
       instant the new round_started_at arrives via WS.
     - 1Hz setInterval calls tickElapsed() which reads Date.now() minus
       the anchor — the formatter is integer ops plus one textContent
       write, cheap enough that there's no point throttling further.
     - formatElapsed: m:ss under an hour, h:mm:ss past one. Matches a
       sports clock — projector view, drama, etc.
     - Defensive: isNaN(Date.parse(...)) check hides the stat instead of
       crashing if the server ever sends garbage. Won't happen with the
       current backend, but the projector is the screen people SEE.

   CSS (projector.css): #proj-elapsed gets font-variant-numeric:
   tabular-nums so the seconds digits don't jiggle as glyph widths change
   each tick. Same trick already used on .rank-num and .rank-score.

   Verified:
     a. End-to-end test: created a 3-player onsite tournament, snapshot
        before round 1 had round_started_at = None (timer hidden); after
        start_next_round the snapshot returned an ISO-T string within
        1s of wall clock.
     b. Round transition: completed round 1 (report + confirm), started
        round 2 after a 2-second sleep; the new round_started_at was
        exactly 2s ahead of round 1's. The projector therefore resets to
        0:00 on round change, which is the right behavior.
     c. JS Date.parse round-trip: 2026-05-18T05:48:17Z parses correctly

     d. node --check on the extracted projector.html script block passes.
     e. Python compile-check on tournament.py passes.
     f. State snapshot still returns tournament, players, current_matches,

[DONE] 9. Multiple dashboard themes.

   The brief called this "CSS-only swap" — turned out half-true. About a
   third of the work was a real CSS-only swap (redefining the existing
   palette variables under [data-theme="..."]). The other two-thirds was
   plumbing that had to exist before the swap could work honestly:

     a. The :root palette had 15-ish rgba() literals scattered through
        main.css and projector.css that DIDN'T read from variables. Things
        like `rgba(184,255,92,0.25)` on the projector's leader-row glow,
        the join-form input's `rgba(13,26,20,0.7)` backdrop, the badge
        borders, the body radial-gradient tints, the result-flash radial.
        Re-themed pages would have shown ghost green glows on every
        non-Forest theme without addressing those. So: added a parallel
        set of variables — --accent-rgb, --moss-rgb, --pine-rgb,
        --rose-rgb, --gold-rgb, --bg-0-rgb, --bg-1-rgb (RGB triples kept
        in sync with the hex variables), --grain-rgb / --grain-alpha for
        the body grid texture, --btn-primary-ink for primary-button text
        contrast, --lime-hover for the hover swap-in, --bg-deeper for the
        body gradient base. Then rewrote every theme-bound literal as
        `rgba(var(--*-rgb), α)`. Generic shadows (rgba(0,0,0,...)) and
        whites kept literal — they're not theme-dependent.

     b. base.html needed an inline <head> script that reads the saved
        theme from localStorage and sets data-theme on <html> BEFORE
        main.css loads. Without this, every page would paint with
        Forest's defaults for one frame and then snap to the chosen
        theme — exactly the flash the inline-script pattern is for. The
        script also flips <meta name="theme-color"> in the same step,
        because the mobile browser chrome reads that during initial parse
        too. Whitelisted theme names (`{forest:1, royal:1, crimson:1,
        ivory:1}`) so a poisoned localStorage value can't apply an
        arbitrary attribute. Wrapped in try/catch because private-mode
        Safari throws on localStorage access, and an uncaught throw in
        <head> kills the rest of the page.

   The four themes shipped:
     - Forest — the original. :root defaults; [data-theme="forest"] has
       zero declarations and is byte-identical to before this task.
     - Royal — deep navy backdrop, warm amber accent. The classic chess
       club aesthetic.
     - Crimson — near-black + signal red. Broadcast / drama mode.
     - Ivory — cream surfaces, charcoal ink, bottle-green accent. For
       daylight-lit rooms and the unusual case where a TV/projector isn't
       in a dark room. Required two extra overrides the dark themes
       didn't: --grain-rgb flips to 0,0,0 with a higher alpha (dark grid
       on cream needs more contrast than the inverse), and
       --btn-primary-ink flips to cream so the dark-green primary button
       has readable text.

   common.js gained EP.THEMES, EP.THEME_BG, EP.getTheme(),
   EP.applyTheme(name). applyTheme() sets the attribute, writes
   localStorage, flips the meta tag, and dispatches an 'ep:theme' window
   event for in-page listeners. Also: a top-level `storage` event listener
   on every page that catches changes from other tabs (the storage event
   doesn't fire in the tab that wrote the value, so this is the inheritor
   side). Net effect: pick Royal on the host dashboard while the
   projector is open on a TV, and the projector re-themes within ~500ms
   without a reload. Verified end-to-end with two browser tabs sharing a
   context.

   host.html got a theme-picker card with four swatches. Each swatch is
   a small linear-gradient preview of that theme's bg + accent —
   hardcoded colors there are intentional (they're previews of OTHER
   themes, so they can't read from the live theme). Picker uses :has()
   for the checked-state styling in modern browsers, with a parallel
   .is-checked class for older Safari; the init code keeps both in sync.
   Subscribes to 'ep:theme' to reflect cross-tab changes back into the
   picker UI.

   default), but the inline anti-flash script was setting it. So an

   Projector view (and every other page) inherits all of this through

[DONE] 10. "Just happened" rank-change ticker copy. ("Alice climbed from #7
    to #3").

    Backend (tournament.py):
      - New _current_ranks(conn, tid) helper: returns {player_id: rank} using
        the same ORDER BY as list_players (score DESC, buchholz DESC,
        sonneborn_berger DESC, elo DESC, name ASC). Re-running the SQL sort
        instead of re-implementing it in Python guarantees the ticker copy
        agrees with what the projector leaderboard actually shows — float-
        equality on buchholz between two languages was the trap to avoid.
      - _finalize_match snapshots ranks BEFORE applying the score update
        (this is what the room is looking at right now), then snapshots
        AFTER _recompute_tiebreaks (this is what they're about to see).
        For every player whose rank actually moved, emits a 'rank_change'
        event with message "X climbed from #N to #M" / "X dropped from
        #N to #M" and a payload of {player_id, name, from_rank, to_rank,
        direction}.
      - Cap of 4 rank_change events per finalize. With 32 players and a
        Buchholz cascade, in pathological cases a single result can shift
        many ranks via tiebreak recompute (Buchholz reads opponent scores,
        and an opponent's score just changed). Sorted by |delta| DESC so
        the biggest movements survive the cap.

    Design decision worth flagging: rank changes are emitted for ALL
    players who moved, not just the two in the match. That includes
    "Carol climbed from #3 to #2 after losing to Alice" — which happens
    because Carol's Buchholz jumps when Alice's score does (her one
    opponent now has more points). This is mathematically correct per
    FIDE Buchholz, and matches what the projector leaderboard will show.
    The alternative — silently moving Carol up without an explanation —
    is worse: spectators look at the leaderboard, see Carol jumped,
    have no idea why. A chess club host running Swiss with Buchholz
    tiebreaks will read "Carol climbed" as exactly the right thing: a
    strength-of-schedule shift. Honest > snappy.

    Frontend (projector.html):
      - renderTicker's filter now includes kind === 'rank_change'.
      - addTickerItem reads payload.direction (parsing the JSON string
        from the server, falling through if malformed) and picks '↑' for
        up, '↓' for down. Existing icons unchanged: ♟ for result, ◆ for
        round_start, + for join.
      - Stagger and 22s auto-removal apply uniformly — a result event
        and its rank_change events appear in sequence with 1.5s offsets,
        which reads like a broadcast (the play, then the standings
        consequence).

    Schema: no changes. New events flow through the existing events
    table; the projector dedupes by event id; the WebSocket broadcast
    layer carries them without modification.

[DONE] 11. Color balance refinement in Swiss. The pre-task-11 _assign_colors
    looked only at the cumulative (whites, blacks) totals — it had no notion
    of "this player just played white twice in a row, give them black now".
    The cumulative rule is symmetric around 0, so two players with equal
    rounds-played almost always have equal whites, and the tiebreaker chain
    fell straight to lexicographic id. That's where the 3-in-a-row sneaks
    in.

    Baseline measurement (n=16, 12 rounds, white-wins-always): 9 of 16
    players experienced a 3+ same-color streak, 2 hit a 4-streak, and the
    field accumulated 11 separate 3+ extensions. n=32, 9 rounds, white-
    wins: 16 of 32 players, 4 hit 4-streaks, 2 hit 5-streaks, 22 total
    extensions. The pre-fix code was producing the bad pattern reliably.

    Fix (pairing.py):
      - New _color_streak(player_id, past_matches) helper. Walks past_matches
        from newest to oldest, counts how many trailing non-bye games were
        the same color. Byes are neutral (skipped, do not reset the
        streak) — FIDE convention, since the bye-getter has no color.
        Returns (color, length); (None, 0) when the player has no games.
      - _assign_colors gains a priority-1 streak check above the existing
        cumulative-balance rule:
          - If exactly one player is on a same-color streak of ≥2, they
            get the OPPOSITE color.
          - If both are on streaks of opposite colors (one is WW, the
            other is BB), each gets their preferred opposite — perfectly
            compatible, assignment is forced.
          - If both are on streaks of the SAME color (both WW), one of
            them MUST extend to 3 (no color assignment can prevent this).
            Falls through to cumulative balance — whoever has fewer total
            whites takes the unwanted extension, which keeps |whites -
            blacks| from drifting on top of the unavoidable streak.
        Rules 2-4 (cumulative whites, then cumulative blacks, then lex
        id) are byte-identical to before the task. When neither player has
        a 2-streak — the common case in any tournament under 4 rounds, and
        the majority case generally — the function returns the same answer
        as the pre-task-11 version.

    Why "above" not "as a tiebreaker": the cumulative rule almost never
    leaves a tie that the streak check could break, BECAUSE the cumulative
    rule is symmetric — equal rounds played mostly means equal whites. The
    streak check has to be ABOVE the cumulative check to fire at all in
    the cases that matter. Verified empirically: on the n=16, 12r config,
    moving the streak check below cumulative changes nothing (≥98% of the
    time the cumulative chain doesn't tie); above cumulative, it fires
    routinely.

    Why not change the matching engine too: the remaining 3-streaks after
    this fix are all "forced" — both players in the pairing were already
    on a same-color 2-streak before the round started. To eliminate those,
    the pairing engine itself would need to consider color compatibility
    when choosing partners (FIDE's full Dutch system does this), which
    means rewriting _max_matching to score candidates by color-conflict
    cost on top of legality. That's a much bigger change with real
    regression risk for task #2's matching invariants. The brief explicitly
    flagged this as "school clubs won't notice"; the right scope is
    catching every AVOIDABLE 3-streak via color assignment alone.

    Round-robin's _assign_colors_rr is untouched. It already alternates via
    _last_color (rule 2 in its priority order), which implicitly avoids
    streaks for the round-robin schedule's pairing structure. No reason to
    muddle two algorithms when one already gets it right.

    Random's pair_random uses _assign_colors, so it inherits the fix for
    free. No code change needed in pair_random.

    Verified (test_color_streaks.py, 8 cases):
      a. _color_streak: empty history (None, 0); one game; two-W ("white",
         2); WBW reset by the B ("white", 1); WWB the most recent is the
         streak ("black", 1).
      b. _color_streak with bye neutrality: W-bye-W is ("white", 2);
         WW-bye-W is ("white", 3); a lone bye is (None, 0).
      c. One-streak handling: p1 on WW vs p2 not on a streak → p1 gets
         black. Symmetric in arg order. Mirror with p1 on BB → p1 gets
         white.
      d. Opposite streaks: p1 on WW, p2 on BB → p1 black, p2 white. Forced
         and compatible.
      e. Same-color streaks (forced extension): p1 on WW, p2 on WW. Falls
         through to cumulative; equal whites + equal blacks → lex id → p1
         gets white. Deterministic across arg order.
      f. Regression (no streaks): empty history; one round each both
         white; one-W vs one-B. All match pre-task-11 behavior exactly.
      g. End-to-end on stressful configs: n=16/12r/white-wins, n=12/7r,
         n=8/7r, n=16/12r/draws — every 3+ extension that occurs is
         provably forced (both players were on the same-color 2-streak
         before the round). Zero avoidable 3-streaks across all four
         configs.
      h. Quantitative reduction: on n=16, 12r, white-wins, the pre-task-11
         code produced 18 separate 3+ extensions across the field; the new
         code produces 8 — all 8 forced. ~56% reduction on the canonical
         stress config.
