[BUG] B2. Swiss bye-recipient chosen before pool pairability check.
  File: app/services/pairing.py:238
  Task 2 invariant: "zero emergency byes when a perfect matching exists."
  Codex repro: with 5 players, past pairs
    {p00-p02, p01-p03, p03-p00, p01-p04, p00-p04, p02-p01}
  and scores p00=3, p01=1.5, p02=1.5, p03=2, p04=1, pair_swiss returns
  p04-p03 plus EMERGENCY BYES for p00, p02, p01. A valid one-bye round
  exists: bye p02, pair p00-p01 and p03-p04.
  Root cause: bye recipient is chosen by the Task 6 selector
  (bye_count, score, elo) without considering whether the remaining
  n-1 pool admits a perfect matching against the rematch graph.
  Fix sketch: when n is odd, try each candidate bye recipient in the
  Task 6 priority order, and accept the first whose REMOVAL leaves the
  remaining pool with a perfect matching (via _max_matching). Fall
  through to the current behavior only if no candidate works.
  Test to add: the Codex repro case in test_swiss.py, plus a
  regression that the Task 6 bye-fairness ordering is preserved
  whenever the priority-1 candidate is feasible.

[BUG] B3. Bye score updates skip _recompute_tiebreaks.
  Files: app/services/tournament.py:197 and app/services/tournament.py:260
  Task 4 invariant: tiebreaks are recomputed on every finalize. But
  the bye-score paths bump the bye'd player's score without calling
  _recompute_tiebreaks. Any previous opponent of the bye'd player has
  stale Buchholz/SB until some later non-bye match finalizes for that
  opponent.
  Fix: call _recompute_tiebreaks at both bye-update sites. Same trigger
  as _finalize_match.
  Test to add: 4-player Swiss, round 1 normal, round 2 awards a bye
  to a player whose round-1 opponent has confirmed games; verify the
  opponent's Buchholz/SB reflect the bye'd player's new score
  immediately after the bye finalizes, not only after a later match.

[DONE] B4. report/confirm now verify {tid} matches the match's tournament.
  Files: app/routers/api.py, tests/test_cross_tournament_match.py
  Added _require_match_in_tournament(tid, mid) helper in api.py alongside
  the existing _require_host helper. It runs a single
    SELECT 1 FROM matches WHERE id = ? AND tournament_id = ?
  and raises HTTPException(404, "Match not found.") on miss. Wired into
  both the report and confirm routes as the first line of the handler,
  before svc.report_result / svc.confirm_result.

  Why router-level and not service-level: the two affected service
  functions (report_result, confirm_result) take only match_id; every
  other match-mutating service function in tournament.py already takes
  tid as a parameter (set_match_table, host_resolve_match) and does its
  own ownership check via "WHERE id = ? AND tournament_id = ?". Pushing
  the guard into the service would mean changing the signature of these
  two functions and every caller — a wider blast radius for no extra
  defense, since the router is the only entry point. Same pattern A3
  and B1 chose: status checks in service, route-shape checks in router.

  Why 404 not 400: the existing convention in report/confirm is 404 for
  res is None (match-shaped problems) and 400 for "error" in res
  (business-logic problems). A tid mismatch is a match-shape problem —
  the match this tid+mid pair identifies doesn't exist. 404 also avoids
  leaking the fact that the mid IS valid in some other tournament.

  Added the database import to api.py so the helper can hit SQLite
  directly without forcing a new service function for one query.

  Tests: tests/test_cross_tournament_match.py — 6 cases.
    - cross-tournament report returns 404 (the headline regression);
    - cross-tournament report does NOT mutate the real match row

      the raw row directly via app.database.db, same helper style as
      test_dispute_resolve.py and test_bye_no_table.py);
    - cross-tournament confirm returns 404 (sets up a legitimately
      reported match first, then attempts confirm via wrong tid);
    - request under a tid that doesn't exist at all also 404s (same
      code path — guards against a future refactor that special-cases
      "tournament exists but match is elsewhere" vs "tournament missing"
      and reintroduces an information leak);
    - legitimate report under correct tid still works (200, status

[DONE] A3. Disputed matches can bypass host resolution.
  Files: app/services/tournament.py, tests/test_dispute_resolve.py
  report_result() previously rejected only status in ('confirmed','bye'),
  so a 'disputed' match fell through and the UPDATE silently reset it
  back to 'reported' — at which point confirm_result() would finalize
  it without ever touching host_resolve_match. The fix adds an explicit
  second guard: status='disputed' returns
    {"error": "Match is disputed — only the host can resolve it."}
  Two branches (rather than folding 'disputed' into the existing
  "already finalized" message) so each rejection states the actual
  situation — a disputed match is the opposite of finalized.
  Other paths re-verified:
    - confirm_result already guards "status != 'reported'", so the
      confirm route was never reachable on a disputed match. The A3
      bug existed *only* because report_result reopened the door by
      flipping disputed -> reported.
    - host_resolve_match still accepts 'disputed' (correct — it's the
      one path that must work).
  Tests: tests/test_dispute_resolve.py — 5 cases.
    - re-report on disputed match returns 400 with "host" in detail;
    - re-report does NOT mutate status/result/reported_by (verified by
      reading the raw row directly via app.database.db);
    - symmetry: the *disputing* player also can't backdoor a result;
    - confirm on a disputed match still rejects (documents the
      confirm_result invariant so a future refactor of that guard
      doesn't silently reintroduce A3);
    - host resolve transitions disputed -> confirmed cleanly with
      confirmed_by='host'.
  Verified: 5/5 pass against the fix; 3/5 fail against the original
  buggy report_result (the two passing pre-fix are the confirm-guard
  and host-resolve tests, which exercise paths that were never
  broken — they document the surrounding invariants).

[DONE] A2. XSS via player name in host.html inline onclick.
  Files: templates/host.html, tests/test_xss_host.py
  Removed every inline event handler (onclick, onchange, onkeydown)
  from the rendered HTML in host.html. The standings remove-button,
  the host-resolve override buttons, and the onsite table-number
  inputs now use data-action / data-pid / data-mid / data-name /
  data-result attributes and are wired via delegated listeners on
  the stable #standings and #matches containers (which survive the
  innerHTML re-renders).
  Names (and IDs) still pass through EP.escape, but the critical
  change is that they now land in HTML attribute-value context, not
  inside a JS string literal. The HTML-escape-then-decode trap is
  structurally impossible once no value sits inside a '...' or
  "..." that the JS parser will read.
  Cleanup: the four `window.X = X` shims at the bottom of the
  <script> block (removePlayer, hostResolve, setTable, tableKey)
  were dropped — they only existed to make the inline handlers
  resolve, and aren't needed now.
  Tests: tests/test_xss_host.py — 6 cases.
    - no inline onXxx= handlers anywhere in the template (the
      structural invariant);
    - no player-name / opponent-name field interpolated inside a
      JS string literal (regex with a (?<!=) lookbehind so HTML
      attribute values don't false-positive);
    - same check for ID fields;
    - positive assertions that the data-action / data-pid /
      data-name / data-mid / data-result attribute names the
      delegated listener depends on are actually present;
    - a unit test that simulates the post-fix rendered HTML for
      hostile names ()  );alert(1);//  ,  <script>...</script>  ,
      <img onerror=...>, mixed-case <ScRiPt>, attribute-breakout

  Verified: 6/6 pass against the fix; 5/6 fail against the original
